LiveDB Demo

SQLAlchemy writes + LiveDB reads — click any endpoint to call it

How this works: LiveDB reads from SQLite via DuckDB's scanner. SQLAlchemy writes to SQLite directly. Two separate connections to the same database — no conflicts.
Open app.py and write_routes.py in VS Code to see LiveDB previews on the db.* lines.
Read Endpoints — LiveDB Previews Work

Active users aged 18+. Variables resolved from function parameter defaults.

Single user lookup. user_id: int = 1 → Alice Smith.

Write Endpoints — Code Time vs Run Time
POST/api/orders/create reads first ✓
GOOD pattern. Reads user via LiveDB first, then writes order via SQLAlchemy. LiveDB preview shows existing data. Click multiple times to see order count grow.
POST/api/users/3/deactivate write before read
BAD pattern. SQLAlchemy deactivates user, then LiveDB reads. At code time, preview shows OLD status. At run time, shows "inactive" because write was committed first.
POST/api/users/3/deactivate-broken uncommitted
WORST pattern. SQLAlchemy writes but never commits. LiveDB sees stale data in both contexts. This is how DB isolation works — uncommitted changes are invisible to other connections.
POST/api/users/3/activate fix ✓
Reactivate user 3 (undo deactivate). Commits first, then LiveDB reads updated status.
